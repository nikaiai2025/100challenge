# 実行計画（PoC）: 青空文庫20冊 / 比較検索UI

## 目的と狙い
本PoCは「検索できるか」ではなく、ユーザーが直感的に「どの検索方式がどう有効か」を判断できる状態を作ることが目的です。  
1つのクエリで以下3方式の結果を並列表示し、差分を見せます。

1. 全文検索（Meilisearch）
2. Google Gemini File Search のみ
3. Google Gemini File Search + 固定DBのフェデレーション検索

特に「位置情報の有無」を体験的に示し、実運用での併用が合理的であることを理解できる状態を狙います。
全文検索：必ずページ数とセットで表示可能
Google Gemini File Searchのみ：ページ数を出す努力（各チャンクにページ数を記載）するものの、決定論的に出ることを保証できない
Google Gemini File Search + 固定DBのフェデレーション検索：メタデータを活用した絞り込み機能を使える（著者・ジャンル・発行年）


## 対象データ
青空文庫の著作権切れ作品 100冊

## 制約・前提
1. Google Gemini APIは無料Tierのみを使用
2. 無料Tierではデータが学習に利用される可能性があるため、公開データのみを投入
3. File Search の検索ではリンクや位置情報が保証されない
4. 物理ページは存在しないため、PoCでは「擬似ページ」を採用

## 擬似ページ仕様（PoC前処理）
1. LLM前処理で一定行数ごとにページを割り当てる
2. 1ページ = 40行
3. 各ページ先頭にメタ行を挿入
4. メタ行の例
   DOC_ID:aozora-0001 | PAGE:12 | LINES:441-480

## アプリの概要（比較UI）
1. 1つの検索ボックス
2. 結果は3カラム固定で並列表示
3. 各カラムの表示項目は統一
4. 表示例
   作品名 / 著者 / 位置（p.xx or 位置不明）/ スニペット / 参照リンク

### 差分の見せ方
1. 全文検索は「位置」を明確に表示
2. File Search は「位置不明」ラベルを明示
3. フェデレーション検索は位置が分かる場合のみ表示

## リスクと対策
1. File Search のリンク/位置が取れない
   対策: UIで「位置不明」を明示し差分として見せる
2. 無料TierのRPM制御（100件投入時）
   対策: Document投入時にレート制限（例: Gemini 1.5 Flashの無料枠なら15 RPMなど）に引っかかる可能性が高いため、インデックス構築スクリプトに適切なスリープ処理（遅延）やリトライ機構を組み込む。またクエリ実行時もキャッシュを活用する。
3. 擬似ページの妥当性
   対策: PoCと割り切り、UIで「擬似ページ」と明示

## Google Gemini File Search 最新ドキュメント反映（人が行う対応事項）
以下は実装前に人が判断・準備すべき事項。

1. File Search Storeの設定
   - API（またはコンソール）を用いてStoreの概念となる準備を行い、表示名を決める。
   - ※Google AI Studioの仕様上、機能名は File Search として提供されています。
2. ドキュメント投入方法の選択
   - API経由で直接アップロード・チャンク送信するか、File APIで別途アップロードしたファイルを紐づけるかを決める。今回は独自メタ行との連動を考慮。
3. メタデータ設計と付与
   - `customMetadata` は1文書あたり最大20件。
   - year/author/genre を必須として付与し、フェデレーションで `metadataFilter` に使う。
4. ドキュメント識別子の設計
   - Document IDは英小文字/数字/ハイフンのみ、長さ40文字以内。
   - displayNameは最大512文字。必要に応じて表示名を設計。
5. Chunkingの方針確認
   - 既定のチャンク分割を使うか、`chunkingConfig` を明示してページとメタ行が分断されないようにするかを決める。
6. 無料Tierのレート制限把握
   - RPM/TPM/RPDの上限がモデルごとにあるため、
     PoCで使うモデルの上限を確認し、レート制御・キャッシュ方針を決める。

## Google Gemini File Search APIインターフェース（要点）
最新ドキュメントに基づく実装要点を整理。

### 1. Store/環境作成
- 用途: アップロードされたファイルを検索できるようにCorpus/Storeの箱を準備する。

### 2. Document登録/投入
- 用途: アップロードしたファイルにメタデータを付与して検索インデックスへ紐づけ。
- `customMetadata` は年/著者/ジャンルを必須付与
- `chunkingConfig` を指定する場合はここで設定

### 3. 検索実行（generateContent + tools.file_search）
- `generateContent` に `tools: [{ "file_search": {} }]` などの形式でToolを渡して検索を実行
- `metadataFilter` で対象文書を絞り込み可能

### 4. 返却データ
- 生成テキストに加え、`groundingMetadata.groundingChunks` に
  取得されたコンテキスト（`retrievedContext` 等）が入る
- `title` / `uri` は optional のため、リンクは必ずしも返らない

### 5. レート制限
- モデルごとにRPM/TPM/RPDが異なるため、使用モデルで確認が必要
