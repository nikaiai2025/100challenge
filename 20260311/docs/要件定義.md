# 要件定義（PoC）: 比較検索UI

## 1. 目的
ユーザーが同一クエリに対する検索結果を並列で比較し、  
全文検索 / File Search / フェデレーションの違いを直感的に判断できる状態を作る。

## 2. 画面構成（UI）
### 2.1 レイアウト
1. 上部: 検索クエリ入力
2. 検索ボタン
3. 注意書きセクション
4. 下部: 3列の検索結果表示

### 2.2 3列の内容
1. 全文検索（Meilisearch）
2. Google File Searchのみ
3. フェデレーション検索（File Search + 固定DB）

### 2.3 表示項目（3列共通）
1. 作品名
2. 著者
3. 位置情報
   - 全文検索: p.xx（擬似ページ）
   - File Search: 位置不明
   - フェデレーション: 位置が分かる場合のみ表示
4. スニペット
5. 参照リンク

### 2.4 同一作品の複数ヒット表示ルール
1. 同一作品のヒットはスコア順にすべて表示
2. 50件を超える場合は50件で打ち切り

## 3. 注意書きセクション（学習用途の説明欄）
各検索方式の特徴・メリット・デメリットを簡潔に示す欄を設ける。

### 3.1 全文検索（Meilisearch）
1. 特徴: 文字列一致中心で厳密
2. メリット: 位置が明確に出る、再現性が高い
3. デメリット: 表記揺れや言い換えに弱い
4. 補足: 本PoCではランキング制御を行わず、デフォルトの関連度順で表示

### 3.2 File Search（Google File Search）
1. 特徴: 意味検索（自然文クエリに強い）
2. メリット: 言い換えに強い、曖昧な質問でも引ける
3. デメリット: 位置情報が曖昧、リンクが保証されない場合がある

### 3.3 フェデレーション（File Search + 固定DB）
1. 特徴: メタデータで対象を絞った上で意味検索
2. メリット: 条件指定と発見性の両立
3. デメリット: 実装が増える、前処理やメタデータ整備が必要

## 4. 検索仕様
### 4.1 クエリ入力
1. 検索クエリは1つ
2. 3方式に同時投下

### 4.3 並び順
1. 全列とも関連度順で表示

### 4.2 絞り込み（フェデレーション検索のみ）
1. 年（範囲）
2. 著者（自由入力）
3. ジャンル（自由入力）

※全文検索 / File Searchのみの列では、あえて絞り込みは行わない  
（フェデレーションの特徴を強調するため）

## 5. メタデータ設計
### 5.1 固定DB
1. book_id
2. title
3. author
4. year
5. genre
6. doc_id
7. file_search_id
8. source_url

### 5.2 File Search（customMetadata）
1. year（数値）
2. author（文字列）
3. genre（文字列）

## 6. ジャンル分類
以下5種に統一:
1. 純文学
2. 娯楽小説
3. 童話
4. 詩
5. 随筆

## 7. 擬似ページ仕様（PoC）
1. LLM前処理で1ページ = 40行
2. 各ページ先頭にメタ行を挿入
3. 例:
   DOC_ID:aozora-0001 | PAGE:12 | LINES:441-480

## 8. フェデレーション検索の挙動
1. 固定DBで絞り込み（年/著者/ジャンル）
2. File Searchに同条件を metadataFilter として渡す
3. 結果統合は book_id で重複排除

## 9. 非要件（PoC範囲外）
1. 厳密な物理ページ番号
2. 本番運用のSLAや監視
3. 大規模データ（20冊以外）
