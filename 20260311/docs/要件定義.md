# 要件定義（PoC）: 比較検索UI

## 1. 目的
ユーザーが同一クエリに対する検索結果を並列で比較し、  
全文検索 / File Search / フェデレーションの違いを直感的に判断できる状態を作る。

## 2. 画面構成（UI）
### 2.1 レイアウト
1. 上部: 検索クエリ入力
2. 検索ボタン
3. メンテナンスボタン（検索準備用）
3. 注意書きセクション
4. 下部: 3列の検索結果表示

### 2.2 3列の内容
1. 全文検索（Pagefind）
2. Google File Searchのみ
3. フェデレーション検索（File Search + 固定DB）

### 2.3 表示項目（3列共通）
1. 作品名
2. 著者
3. ページ番号
   - 全文検索: p.xx
   - File Search: 位置不明
   - フェデレーション: 位置不明
4. スニペット

### 2.4 同一作品の複数ヒット表示ルール
1. 同一作品のヒットはスコア順にすべて表示
2. 50件を超える場合は50件で打ち切り
3. 各列の最大件数は50件、ページネーションで表示

## 3. 注意書きセクション（学習用途の説明欄）
各検索方式の特徴・メリット・デメリットを簡潔に示す欄を設ける。

### 3.1 全文検索（Pagefind）
1. 特徴: 静的サイトに対する全文検索（事前インデックス生成）
2. メリット: 位置が明確に出る、再現性が高い、GitHub Pagesで完結
3. デメリット: 表記揺れや言い換えに弱い、インデックス再生成が必要

### 3.2 File Search（Google File Search）
1. 特徴: 意味検索（セマンティック検索）、さらにAI処理しているため自然文クエリに強い(RAG)。
2. メリット: 言い換えに強い、曖昧な質問でも引ける。
3. デメリット: 位置情報が曖昧、リンクが保証されない場合がある
4. 補足: RAGではない意味検索（セマンティック検索）も存在するが、実装の手間があるので今回は対象外とした。Google File Searchを使うとチャンク化等の手間を削減できる。

### 3.3 フェデレーション（File Search + 固定DB）
1. 特徴: 元データに含まれない外部情報（作品メタデータ等）と組み合わせた検索が可能
2. メリット: 柔軟な設定で検索が可能
3. デメリット: 実装が増える、固定DBの準備が必要
4. 補足: 今回のような単純な絞り込み要件は、File Search機能で「CustomMetadata」を付与して検索することも可能。今回はフェデレーション検索＝外部DBと組み合わせることが可能なことを強調するためCustomMetadataは使用していない

## 4. 検索仕様
### 4.1 クエリ入力
1. 検索クエリは1つ
2. 3方式に同時投下

### 4.3 並び順
1. 全列とも関連度順で表示

### 4.2 絞り込み（フェデレーション検索のみ）
1. 年（範囲）
2. 著者（自由入力）
3. ジャンル（自由入力）

※全文検索 / File Searchのみの列では、あえて絞り込みは行わない  
（フェデレーションの特徴を強調するため）
※フェデレーションの絞り込みは固定DB側で行い、File Search側には条件を渡さない

## 5. メタデータ設計
### 5.1 固定DB
1. book_id
2. title
3. author
4. year
5. genre
※固定DBは `doc_id_map.jsonl` を正とし、`doc_id` を `book_id` として扱う。  
`title` は `file_name` から拡張子を除いた文字列を用いる。

### 5.2 File Search（customMetadata）
1. customMetadataは付与しない
※customMetadata はKey-Value形式。1ドキュメントあたり最大20件。

## 6. ジャンル分類
以下5種に統一:
1. 純文学
2. 娯楽小説
3. 童話
4. 詩
5. 随筆

## 7. ページ仕様（PoC）
1. 前処理で1ページ = 40行
2. 各ページ先頭にメタ行を挿入
3. 例:
   DOC_ID:aozora-0001 | PAGE:12 | LINES:441-480
4. ページ番号はUI/検索上は実ページとして扱う
※メタ行はすべてのページに必ず含める（フェデレーションの突合に使用）

## 8. フェデレーション検索の挙動
1. 固定DBの条件（年/著者/ジャンル）で対象の book_id を抽出する
2. File Searchは条件なしで検索する（customMetadata未使用のため）
3. 取得チャンク本文のメタ行から `DOC_ID` を抽出し、1の book_id に一致する結果のみ表示する
4. 表示用の作品名・著者は固定DBの情報を使用する
※`DOC_ID` の抽出はメタ行の先頭文字列 `DOC_ID:` を起点に行う  
※`DOC_ID` が抽出できない結果は表示対象外とする

## 9. その他の要件
1. シンプルな実装（本格的なシステム運用レベルのSLAや監視は不要）
2. Google Gemini File Searchの仕様を参考に実装
https://ai.google.dev/gemini-api/docs/file-search?hl=ja
https://ai.google.dev/api/file-search/file-search-stores?hl=ja#method:-media.uploadtofilesearchstore
3. メンテナンス機能はFile Search Storeへのアップロード・削除のみを対象とする（画面上にボタンを設置）
4. 固定DBは前準備として作成済みであることを前提とする
5. GitHub Pages で公開予定。`C:\Users\user\Documents\PGM連携\100challenge\index.html` からリンクされる前提
6. 全文検索は Pagefind を採用し、事前にページ生成＋インデックス生成を行う  
   - 生成スクリプト: `20260311/server/build_pagefind_site.js`  
   - インデックス生成: `npx -y pagefind --site 20260311`

## 10. API仕様（Googleドキュメント準拠・最小）
### 10.1 検索（File Search）
1. Gemini API の `generateContent` を使用し、`tools` に File Search を指定する
2. 対象の File Search Store 名を指定する
3. 検索条件は File Search 側に渡さない

### 10.2 メンテナンス（File Search Store）
1. Store 作成: `fileSearchStores.create`
2. アップロード: `uploadToFileSearchStore`（完了まで Operations をポーリング）
3. 削除: `fileSearchStores.delete`（`force=true` でStore配下を含めて削除）

## 11. 実装前提（必須）
1. 全てGitHub Pages上の静的サイトとして提供する。Google（FileSearch用のエンベディングされたベクトルDBを提供）を除き、バックエンド用の別サーバーは存在しないものとして構築せよ。
2. Gemini APIキー等の秘匿情報はコミットしない。サーバー上で実行するときはBYOK方式とする（ボタンから登録）
3. 前処理済みテキストは `Aozora_Texts_Preprocessed/` を入力としてインデックス投入に使う
4. Gemini APIキーは `.env` に記載済みの値を使用する

## 12. デバッグ
1. 検索APIのレスポンスはブラウザのコンソールにそのまま出力する
