
## 全文検索（キーワード検索）
- 単語や用語の一致を中心に、タイトルや固有名詞などの厳密な一致において高い信頼性と再現性を持ちます
- ヒットする根拠が「単語が含まれるか」という明快な論理であるため、検索結果の透明性が高いのが特徴です
- アルゴリズムが成熟しており、大規模なデータに対しても計算負荷が低く、安定した高速応答が可能です
- 表記揺れや類義語に対応して精度を制御するためには、独自の辞書（シソーラス）の継続的なメンテナンスが必要です
- 特定の文献をピンポイントで特定したい場合など、予測可能性を重視する用途に最も適しています

### 今回のPJにおける実装方法
- 作品テキストを HTML 化して `20260311/pages/` に配置し、Pagefindで静的インデックスを生成
- ページ生成は `server/build_pagefind_site.js`、インデックス作成は `npx -y pagefind --site 20260311`
- フロント側は `app.js` の `searchPagefind()` で `pagefind.search()` を実行し結果を表示


## セマンティック検索（Google File Search）
- Google File Search のベクトル検索により、クエリの意味に近い断片を抽出する手法です。
- 文字列一致では見つかりにくい言い換え・概念的な近さを拾えるのが強みです。
- 返るのは「検索断片」であり、回答生成は行わないため用途は関連箇所の抽出に限定されます。
- 文書の分割（チャンク化）からベクトル化、保存までを自動で実行するツールとして利用できます。
- 厳密一致や構造的な絞り込みは、キーワード検索やメタデータDBと組み合わせることで補完できます。

### 今回のPJにおける実装方法
- `server/upload_runner.js` / `server/upload_cli.js` で File Search Store を作成し、青空テキストをアップロード
- フロントは `generateContent` に `file_search` ツールを指定し、`groundingChunks` を取得して表示
- 回答生成は行わず、検索断片の比較表示に用途を限定

## RAG（Retrieval-Augmented Generation）
- 検索で取得した断片を根拠に、AIが回答文を生成する手法です。
- 検索結果と回答が同時に得られるため、ユーザーは「根拠」と「結論」をセットで確認できます。
- 回答生成が入る分、セマンティック検索よりも待機時間とコストが増えます。
- 生成結果の品質は、検索で引ける断片の質と量に強く依存します。

### 今回のPJにおける実装方法
- `generateContent` に `file_search` ツールを指定し、同一レスポンス内の `candidates` をRAG回答として表示
- 取得した `groundingChunks` はセマンティック検索・フェデレーション検索にも流用
- 回答は1件のみ表示し、全文はモーダルで確認できるようにしている

## フェデレーション検索（連合検索）
- 意味を広く探すセマンティック検索（Google File Searchなど）と、厳密に探す独自のメタデータDB（キーワード検索など）を同時に走らせて結果を統合するアプローチです
- ベクトル化すると意味がぼやけやすい固有名詞（エラーコードなど）や、日付・カテゴリなどの厳密なフィルタリングを自前DB側で補完できます
- AIモデルが学習していない特殊な業界用語や略称の類義語展開（シソーラス）を自前DBで100%制御できるため、検索漏れを防げます
- 文章の断片やDBの構造化データといった性質の異なる検索結果をAI（Geminiなど）に渡すことで、矛盾を解消した自然で納得感のある回答を生成させることが可能です
- 自動RAGツールの手軽さを享受しつつ、重要なキーワード検索の正確性を自前DBで担保する「良いとこ取り」の戦略として有効です


### 今回のPJにおける実装方法
- `doc_id_map.jsonl` を読み込み、年・著者・ジャンルで固定DB側フィルタを適用
- File Searchの `groundingChunks` から `doc_id` を抽出し、固定DBで絞ったIDのみ残す
- `buildFederationResults()` でメタデータを付与してUIに出力
